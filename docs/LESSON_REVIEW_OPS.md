## 1. 見直しトリガー

### 1-1. ユーザーからの誤り報告

- アプリ内フィードバックフォームから「コンテンツ誤り」カテゴリで受信
- 同一レッスンへの報告が **2件以上** → 即時見直し開始
- 1件でも **問題の正解ミス** に関する報告 → 即時見直し開始
- Jira チケット起票: `SCRUM-XXXX` (Component: Content)

### 1-2. 四半期定期レビュー

- **実施タイミング:** 四半期末の月（3月 / 6月 / 9月 / 12月）第1週
- **対象:** 全レッスン（全ステップ・全問題）
- **担当:** Apollo（CTO の協力を得て実施）
- チェックポイント → [6. レッスン品質チェックリスト] 参照

### 1-3. コンテンツ品質スコア低下

- 品質スコアの定義: レッスン完了率 × 正答率 × ユーザー評価（★）の複合指標
- **トリガー基準:**
  - レッスン完了率が全体平均を **20% 以上下回る**
  - 特定ステップの正答率が **40% 未満**（難易度ミスの可能性）
  - ユーザー評価平均が **3.0 未満**
- 週次モニタリング（Supabase Analytics → Apollo が毎週月曜に確認）

## 2. 見直しプロセス

```
[1. 報告受信 / トリガー検知]
ユーザー報告 / 定期レビュー / スコア低下アラート
      │
      ▼
[2. 内容確認]
該当レッスン・ステップを実際にプレイして確認
誤りの種類を特定（問題文 / 選択肢 / 正解 / 解説 / 難易度設定 / ステップ種別）
Jira チケット起票: Component=Content, 誤り種別を label に設定
      │
      ▼
[3. 修正]
- コンテンツ JSON / DB レコードを修正
- 修正内容を PR に記録（変更前 / 変更後 を明示）
- レビュアー: Apollo（または指定メンバー）
      │
      ▼
[4. SITテスト]
https://logic-sit.onrender.com で修正内容を確認
- 該当レッスンを最初から最後までプレイ
- 修正箇所が正しく反映されているか
- 隣接ステップ・レッスンへの影響がないか
- CTO が SIT環境での動作確認を実施
      │
      ▼
[5. 本番デプロイ]
SIT 確認 OK → main ブランチへ merge → 本番デプロイ
Jira チケット Done → feedback.status を "closed" に更新
ユーザー報告の場合、報告者への御礼通知（任意）
```

## 3. 難易度基準

### 3-1. 難易度定義

| 難易度 | ラベル | 対象ユーザー | 説明 |
|--------|--------|-------------|------|
| 初級 | Beginner | 論理思考初心者 | 日常的な事例を使った具体的な問題。専門知識不要。正答率目標: 70%以上 |
| 中級 | Intermediate | ビジネスパーソン | 複数の要素を組み合わせた推論が必要。ある程度の抽象思考を要する。正答率目標: 50〜70% |
| 上級 | Advanced | 論理思考上級者 | 複雑な多段階推論・批判的思考・前提の見直しが必要。正答率目標: 30〜50% |

### 3-2. 各レベルの問題設計ガイドライン

#### 初級 (Beginner)
- **事例:** 日常生活・身近なシチュエーションを使用（買い物、家族、友人関係など）
- **選択肢:** 4択、明確に誤りの選択肢を含む
- **解説:** 「なぜこれが正解か」を段階的に説明。専門用語は使わない
- **ステップ数:** 3〜5ステップ（短くテンポよく）
- **禁止:** 二重否定、複数条件の複合、抽象概念のみの問題

#### 中級 (Intermediate)
- **事例:** ビジネス・社会的文脈（会議、プロジェクト、意思決定場面）
- **選択肢:** 4択、「惜しい」誤答を含む（一見正しそうに見える選択肢）
- **解説:** 正解の論拠 + 各誤答がなぜ誤りかを解説
- **ステップ数:** 4〜7ステップ
- **活用推奨:** think ステップで「まず自分で考えさせる」設計

#### 上級 (Advanced)
- **事例:** 複雑な社会問題・哲学的問い・ビジネス戦略
- **選択肢:** 4〜5択、全て一定の論拠を持つ選択肢（正解が明確に1つ）
- **解説:** 思考プロセス全体を丁寧に説明。反論への対応も含む
- **ステップ数:** 6〜10ステップ
- **活用推奨:** case ステップで実際のケーススタディを提示

## 4. 新ステップタイプ使用ガイドライン

### 4-1. think ステップ

**用途:** ユーザーに自分で考える時間を与える「立ち止まり」ステップ

**使用シーン:**
- 問題提示後、すぐに選択肢に進む前に「考えてみよう」を促す
- 複雑な概念の理解確認（知識問題ではなく内省を促す）
- 議論の前提整理（「あなたはどう思う？」）

**設計ルール:**
1. **問いかけは1つだけ** — 複数の問いを詰め込まない
2. **オープンエンド** — 正解を求めない、思考プロセスを促す
3. **次ステップとの接続** — think で考えた内容が次の問題に活きる設計
4. **所要時間の目安:** 30秒〜2分で回答できる問いにする

**例:**
```
「この状況で、あなたならどんな情報が欠けていると感じますか？
少し考えてみてね。」
→ 次ステップ: 「情報の欠如」を扱う問題
```

### 4-2. case ステップ

**用途:** 実際のケーススタディや具体的なシナリオを提示するステップ

**使用シーン:**
- 上級レッスンの導入部（複雑な状況設定）
- 中級レッスンでの「実際の事例」紹介
- 複数の問題が同一ケースを参照する場合の冒頭

**設計ルール:**
1. **ケースは完結している** — 必要な情報は全てこのステップに含める
2. **長さの上限:** 300文字以内（長すぎると離脱率が上がる）
3. **登場人物は最小限** — 2〜3名まで。名前は覚えやすいもの（例: 田中、鈴木）
4. **時制は現在形** — 没入感を高めるため「〜している」「〜の状況だ」
5. **後続ステップとの紐付け** — case の情報が直接問題に使われる設計

**例:**
```
「田中部長は部下の佐藤さんに「このプロジェクトはうまくいっている」と言いました。
しかし佐藤さんのレポートには遅延の記録が残っています。」
→ 次ステップ: 矛盾を発見する問題
```

## 5. レッスンの品質チェックリスト

定期レビューおよびリリース前チェックに使用する。

### 5-1. コンテンツ正確性

- [ ] 問題文に誤字脱字がない
- [ ] 正解が実際に正しい（論理的に検証済み）
- [ ] 誤答の選択肢が一定の根拠を持ちつつ誤りである
- [ ] 解説が正解の論拠を明確に説明している
- [ ] 解説が各誤答の「なぜ誤りか」を説明している（中級以上）
- [ ] 参照する事例・情報が最新かつ正確である

### 5-2. 難易度適切性

- [ ] 難易度ラベルがガイドラインの定義と一致している
- [ ] 想定正答率の範囲内に収まっている（実データで確認）
- [ ] 初級: 専門用語を使用していない
- [ ] 上級: 複数段階の推論が必要な設計になっている

### 5-3. UX・可読性

- [ ] 各ステップが300文字以内（case ステップを含む）
- [ ] 一貫したトーン・語尾（〜だよ / 〜だね 系）
- [ ] ステップ間のつながりが自然（唐突な話題転換がない）
- [ ] 最後のステップで達成感・まとめがある

### 5-4. 新ステップタイプ（該当する場合）

- [ ] think ステップ: 問いかけが1つだけ / 次ステップと接続
- [ ] case ステップ: 300文字以内 / 後続問題で活用されている

### 5-5. 技術確認（SIT環境）

- [ ] レッスンが最初から最後まで正常に進行する
- [ ] 全選択肢がタップ可能（UI不具合なし）
- [ ] 進捗 / スコアが正しく記録される
- [ ] 前後のレッスンへの遷移が正常
