# 承認フロー — Logic プロジェクト

## 原則

「オーナーが承認するまで動かないもの」と「Apolloが自由に動けるもの」を明確に分ける。

---

## オーナー承認が必要なもの（必ず止まって確認）

| 分類 | 具体例 |
|------|--------|
| 本番デプロイ | `main` へのマージ・Renderへの手動デプロイ |
| 課金・Stripe操作 | 価格変更・返金・プラン廃止 |
| ユーザーデータ操作 | データ削除・バルクアップデート |
| 外部サービス契約 | 新規SaaS導入・有料プラン変更 |
| RULES.md の変更 | どのエージェントのルールも変更不可 |
| 予算を伴う判断 | API費用が月1万円を超える設計変更 |

**承認方法**: チャットで「〇〇をしてよいですか？」と確認し、「やって」「OK」などの明示的な返答を得てから実行する。

---

## Apollo が自由に動けるもの（確認不要）

| 分類 | 具体例 |
|------|--------|
| SITへのデプロイ | `develop` ブランチへのpush → logic-sit 自動デプロイ |
| コード実装・修正 | バグ修正・新機能開発（SIT止まり） |
| Jira操作 | チケット起票・ステータス変更・コメント |
| エージェントファイル更新 | KNOWLEDGE.md・LOG.md の更新 |
| 調査・分析 | 技術調査・競合調査・コスト試算 |
| ドキュメント作成 | 設計書・要件定義・READMEの作成・更新 |
| Supabaseマイグレーション（SIT） | SIT環境のみ |

---

## 判断に迷ったとき

以下の順で考える：

1. **本番に影響するか？** → Yes なら承認必要
2. **お金が動くか？** → Yes なら承認必要
3. **ユーザーデータに触るか？** → Yes なら承認必要
4. **上記すべてNo** → 自由に動く

---

## 緊急時（本番障害）

1. まずオーナーに状況を報告
2. 報告後5分以内に返答がなければ、被害最小化の対処を先行して実施
3. 対処内容を事後報告する
