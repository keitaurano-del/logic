# CXO 権限委譲・業務委任設計

## 問題認識
Apollo（CEO）が全部自分でやっている。CXOが名前だけの存在になっている。
CEOの仕事は「自分がやる」ではなく「誰にやらせるか決めて、結果を確認する」こと。

---

## 各CXOの責任範囲と定期タスク

### CTO — 技術責任者
**オーナー**: 技術的な判断・実装品質・インフラ

| 定期タスク | 頻度 | 内容 |
|---|---|---|
| SIT動作チェック | 毎日 | 全画面を自動or手動で確認。バグ発見→Jira起票 |
| ビルド状態確認 | 毎日 | Render/GitHub Actionsのステータス確認 |
| 技術負債レビュー | 週1 | コードの問題点をリストアップ→改善提案 |
| APIコスト監視 | 週1 | Supabase/Claude APIの使用量チェック |

**委任する判断権**:
- バグ修正の優先順位（P0/P1は即対応、P2以下は週次で判断）
- SIT環境のデプロイ判断
- リファクタリングの範囲決定
- ライブラリ選定・技術調査

### CPO — プロダクト責任者
**オーナー**: ユーザー体験・機能設計・コンテンツ品質

| 定期タスク | 頻度 | 内容 |
|---|---|---|
| ユーザーフィードバック確認 | 毎日 | Supabaseのfeedbackテーブルを確認→分類→Jira起票 |
| レッスン品質レビュー | 週1 | 既存コンテンツの改善点を洗い出し |
| 機能リクエスト整理 | 週1 | 要望を集約→ロードマップに反映 |
| 競合ウォッチ | 週1 | Brilliant/Duolingo等の新機能チェック |

**委任する判断権**:
- レッスンコンテンツの追加・修正
- UI文言の変更
- UXフローの改善提案
- フィードバックの優先度判断

### CMO — マーケティング責任者
**オーナー**: ユーザー獲得・ブランド・グロース

| 定期タスク | 頻度 | 内容 |
|---|---|---|
| X投稿の下書き作成 | 毎日 | 翌日分の投稿案を準備 |
| エンゲージメント分析 | 毎日 | 前日の投稿のインプレッション・リプライ数確認 |
| フォロー候補リサーチ | 毎日 | 同ジャンルのアカウントを5件ピックアップ |
| ASO調査 | 週1 | Play Storeの検索順位・キーワード確認 |
| マーケ施策振り返り | 週1 | 何が効いて何が効かなかったか |

**委任する判断権**:
- 投稿内容・タイミングの判断
- フォロー/リプライ戦略の実行
- マーケティング素材の作成
- グロース施策の提案

### COO — 運用責任者
**オーナー**: 運用品質・KPI・プロセス管理

| 定期タスク | 頻度 | 内容 |
|---|---|---|
| KPIダッシュボード更新 | 毎日 | DAU/WAU, ストリーク継続率, レッスン完了数 |
| Jira棚卸し | 週1 | ステータスが古いチケットの整理 |
| インシデント振り返り | 発生時 | 原因分析→再発防止策→Jira起票 |
| プロセス改善提案 | 月1 | 業務フローのボトルネック特定 |

**委任する判断権**:
- KPIの異常値検出→アラート判断
- Jiraチケットのステータス管理
- 運用ドキュメントの更新
- サポート対応フローの設計

## Apolloの仕事（CEOとして）

### やること
- CXOへのタスク指示と結果レビュー
- 週次でCXOレポートを確認→フィードバック
- Keita-san（会長）への報告・相談
- 戦略レベルの意思決定
- CXO間の調整・優先順位の最終判断

### やらないこと（CXOに委任）
- コードの直接修正 → CTO
- レッスンコンテンツの作成 → CPO
- X投稿の作成 → CMO
- KPI集計 → COO
- バグの直接調査 → CTO

---

## 実装方法: cronによるCXO定期稼働

### Phase 1（今すぐ）
CXOごとにcronジョブを作成。各CXOは自分のAGENT.md/KNOWLEDGE.md/RULES.mdを読み込んで、定期タスクを実行。結果をSlack or チャットに報告。

| CXO | cron | 内容 |
|---|---|---|
| CTO | 毎日 06:00 JST | SITチェック→バグ報告 |
| CPO | 毎日 07:00 JST | フィードバック確認→整理 |
| CMO | 毎日 06:30 JST | 当日のX投稿案作成 |
| COO | 毎日 08:00 JST | KPI集計→日報 |
| Apollo | 毎日 09:00 JST | CXOレポート確認→週次まとめ |

### Phase 2（来週〜）
- CXO間の連携ルール策定（例: CTOがバグ見つけたらCPOにUX影響を確認）
- 週次CXO会議（Apollo主催、全CXOのレポートを集約して判断）

### Phase 3（将来）
- CXOの判断履歴を蓄積→KNOWLEDGEに反映→判断精度向上
- 新しいCXO追加（CFOなど）

## 成功指標
- Apolloが直接コード修正やコンテンツ作成をしなくなる
- CXOレポートの質が上がっていく
- Keita-sanへの報告が「やったこと」ではなく「判断したこと」中心になる
