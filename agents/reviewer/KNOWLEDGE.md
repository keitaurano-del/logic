## 再発要警戒パターン

### v3 ダークテーマ関連の白抜け（2026-04-27 BUG-002〜004）
- **症状**: 画面の上部 / 下部 / ヘッダー / safe-area が白く抜ける
- **真因**: `<html>` の class が `mode-light` のままで、`<body>` の `mode-dark` と乖離
- **チェック観点**:
  - QA が新規画面のテストで「画面の端 / セーフエリア」のスクショを撮っているか
  - ボトムナビ下端、ヘッダー上端、横スワイプ時の余白を必ず確認させる
  - `getComputedStyle(document.documentElement).backgroundColor` の出力をレポートに含めると尚良し

### legacy `[style*="color: #..."]` override（2026-04-27 BUG-001）
- **症状**: 一部画面でデザイナーが意図した濃色文字が白に化ける
- **真因**: `index.css` の attribute selector が inline style を強制上書き
- **チェック観点**:
  - 新規画面で inline `color: '#0F1...'` が使われている場合、override に巻き込まれていないか
  - クリーム / 黄色背景のカードで文字が読めるかを必ず確認

### React inline style と CSS override の競合
- React は inline style を `rgb(...)` 形式で出力 → CSS の `[style*="#"]` セレクタでは捕捉できない場合あり
- 既存の `[style*="rgb(...)"]` 形式の override が「効きすぎ」て他コンポーネントを巻き込むリスク
- **チェック観点**: `index.css` の attribute selector を変更したリリースは特に慎重に検証

## レポート品質チェックの典型パターン

### よくある「甘いレポート」
- 「修正効いてると思います」←  実機確認の証拠なし → REJECT
- 「他の画面は問題なし」← 触ったかわからない → 触った画面を全部明示させる
- 「以前と同じです」← before/after スクショ無し → 比較スクショ要求

### 良いレポートの条件
- 各画面のスクショが PNG ファイルで実在
- 失敗には CSS 実測値（`getComputedStyle` 結果）が添えられている
- 修正後リリースなら before/after の両方が並ぶ
- 過去指摘との照合が文章で明示されている

## v3 デザインの主要 CSS 変数

正しい暗背景：`var(--bg-primary)` = `#082121`
正しい白文字：`var(--text-primary)` = `#F2F7F6`
正しい mint：`var(--accent)` = `#70D8BD`

これらと別の値（特に `#FAFAFB`, `#FFFBF0`, `rgba(245,241,232,...)`）が
画面に出ていたら **legacy 残骸** の可能性大。要修正。

## 連携メモ

### CTO への修正依頼を出す時
- ファイルパスを **絶対パス** か `src/` 起点で明示
- 修正方針は2案以上提示（保守性重視 vs 即効性）
- スクショパスを必ず添付

### QA に再テスト依頼する時
- 再テスト範囲を画面名 + 確認観点で具体的に
- 新たに追加したいチェック項目があれば明示
- 期限を切る（例: 「次のpush後30分以内」）
